Four Steps and 40+8 Steps Do Not Share a Unit

A field guide to comparing video-generation schedules without inventing a benchmark

machine-learning
Author

Vaibhav Nakrani

Published

July 11, 2026

When one video model says “four steps” and another says “40 base steps plus 8 refiner steps,” what exactly are we comparing?

The tempting answer is a stopwatch model: one step is one identical tick, so fewer ticks means proportionally less work. That model is useful only inside one implementation. Across implementations, step is a local counter. It tells us that something advanced, but not yet what advanced or how much computation happened.

The goal here is to build a better mental model from one small example, then use it to read three real systems without inventing a shared unit.

What can one step mean?

Start with a video generator’s current state: the working representation it is changing. In diffusion systems this is often a compact numerical array called a latent, but for now it is enough to picture a rough eight-frame clip.

A schedule update takes that current state and produces the next state. The scope of the update says which object moves forward: the whole clip, one stage of a pipeline, or the next frame or block of frames. An actual model call, also called a network invocation, is one execution of a neural network. One nominal schedule update may map to one call, several calls, or a call whose workload differs from another system’s call.

That last sentence is the key: a nominal step is a counter defined by its own implementation. Its mapping to network calls and work has to be reported or measured.

Yellow means the call mapping is still unknown on every path. A node shows one nominal update or one stage counter; it does not promise one identical network call.

Path A’s 4 counts four successive revisions of the entire clip. Path B has two separate counters that both read 4: one belongs to the base stage and one to a later refiner, which is a second stage that improves the base result. Path C’s 4 counts four temporal advances; each advance appends a two-frame block, so the four blocks cover the fictional eight-frame output. That toy arithmetic illustrates the scopes; it is not a claim about any cited model.

Pause here and predict: if every yellow node happened to require one call, would the calls be comparable? Not necessarily. A call over all eight frames can have a different workload from a call over two new frames, and the base and refiner can be different networks. Call count is a better denominator than the bare word step, but it is still not a runtime benchmark.

Now apply the model to real schedules

With the toy paths in view, the paper-specific terms become easier to place.

Dynamic-in-Few-Step follows the whole-clip pattern. Denoising means repeatedly turning a noisy working state into a cleaner one. Distillation trains a student to reproduce a longer teacher process with fewer scheduled updates. The paper reports a four-step distilled Wan-14B schedule and a 30× speedup relative to its own 50-step teacher (paper).

LingBot-Video follows the two-stage pattern. Its public text-to-video workflow exposes 40 base steps followed by 8 refiner steps (repository). The prompt supplies conditioning: information that guides what the model generates. The two step settings are separate stage controls. The public material does not establish that they equal 48 interchangeable model calls, or that a base call costs the same as a refiner call (model card).

Causal Forcing++ follows the rollout pattern. Autoregressive generation produces a later frame or block while conditioning on earlier generated context. A temporal rollout repeats that operation to extend the video through time. In the paper’s reported implementation, the first latent frame uses four sampling steps; each of the subsequent 20 latent frames uses either one or two sampling steps, depending on the configuration (paper). A latent frame is a frame-like unit in the model’s compact internal representation, so this counter is scoped to a latent-frame advance rather than one full-clip trajectory.

Notice what is missing: there is no “total comparable steps” column. The omission preserves the units rather than hiding them.

Three buckets determine actual work

Once the schedule unit is clear, actual runtime becomes easier to reason about. Put the work into three buckets, one at a time.

1. Dependent updates

An update is dependent when the next update needs the state produced by the previous one. In toy Path A, update 2 cannot revise the clip until update 1 has produced its state. Four dependent updates therefore create four ordered positions even before we know how expensive each position is.

Fewer ordered positions can reduce latency, which is why schedule compression matters. Dynamic-in-Few-Step places its four-step claim in this bucket: its distilled student replaces a 50-step teacher schedule in the paper’s reported setup (paper). The comparison is meaningful against that teacher; it is not a conversion from Dynamic’s steps to LingBot’s stage controls.

2. Work inside each model call

Calls with the same count can perform different work. In the toy example, one call might revise the state of all eight frames, while another processes only a new two-frame block. Network size and the amount of active computation can change the cost again.

Dynamic-in-Few-Step also changes this bucket with timestep-specific structural sparsity: different denoising positions use different subsets of network structure. The paper reports 24% fewer per-step floating-point operations (FLOPs, an estimate of arithmetic work) and a 1.2× wall-clock improvement for that component (paper). The gap between those numbers is a useful warning: less arithmetic does not translate mechanically into the same percentage of elapsed-time reduction.

LingBot’s 30B-A3B label belongs here too. It describes a mixture-of-experts model, or MoE, which routes an input through only part of the network. The label indicates 30 billion total parameters with about 3 billion active, while its 40-step base and 8-step refiner settings remain schedule counters (model card). Routing can change work inside a call; it does not erase the ordered calls.

3. Work outside or between model calls

A generation pipeline can do work before, between, and after its repeated network calls. LingBot’s public path includes a prompt rewriter before generation and a refiner after the base stage (repository). A prompt rewriter expands or changes the user’s text before it conditions the generator. A decoder performs the later conversion from the model’s compact latent state into viewable frames.

Toy Path B makes the boundary visible: handing the base result to a refiner is work that Path A does not necessarily have. Even perfect agreement on model-call counts would leave prompt processing, stage transitions, and decoding to measure separately.

The three-bucket model is now complete:

  1. How many updates must happen in order?
  2. How much work happens inside each model call?
  3. What work happens outside or between those calls?

A reader who keeps those buckets separate can explain why a smaller counter may help without pretending it proves an end-to-end speed ratio.

A compact reporting contract

The comparison still needs a matched job. Fix the output first: width, height, number of frames, frame rate, and conditioning mode. Fix the quality test next: the reference system, prompt set, evaluator, and acceptable difference. Only then record schedule and runtime.

output:
  width_height: REQUIRED
  frames_and_fps: REQUIRED
  conditioning_mode: REQUIRED

schedule:
  unit_advanced: REQUIRED        # full clip, stage, frame, or frame block
  nominal_updates_by_stage: REQUIRED
  actual_model_calls_by_stage: UNKNOWN_UNTIL_MEASURED
  outside_call_work: REQUIRED    # e.g. prompt rewrite, refiner handoff, decode

runtime:
  accelerator_and_precision: REQUIRED
  batch_size: REQUIRED
  warmup_runs: REQUIRED
  measured_runs: REQUIRED
  median_end_to_end_time: REQUIRED

quality:
  reference_and_evaluator: REQUIRED
  acceptance_threshold: REQUIRED

Every field has one job. conditioning_mode records whether text, an image, or another input guides generation. The accelerator is the compute device, usually a GPU; precision is the numerical format used for its calculations. batch_size says how many requests are processed together. warmup_runs are unmeasured practice runs that let loading and compilation settle before timing. measured_runs says how many timed repetitions support the median. The median is the middle observed end-to-end time, so one unusually slow run does not define the result. Finally, acceptance_threshold states how much measured quality difference the comparison permits.

UNKNOWN_UNTIL_MEASURED is not a weakness in the schema. It prevents a command-line counter from silently turning into a model-call count. Once instrumentation supplies the mapping, the three work buckets tell us where the elapsed time came from.

The benchmark to run

Suppose all three implementations were available. Ask each for the same output and quality contract, then preserve its native schedule names. Instrument every model boundary and record only four measurements:

Measurement Why it is needed
Nominal updates by native stage Preserves what each implementation actually counts
Actual model calls by stage Reveals the mapping hidden behind the word step
Time inside calls versus outside calls Separates schedule depth from other pipeline work
End-to-end time at matched quality Tests the speed claim the user ultimately experiences

The result might show that fewer dependent updates dominate runtime. It might instead show that heavier calls, repeated frame-block rollout, refinement, or decoding absorbs much of the gain. Either outcome answers an engineering question that a naked step count cannot.

The corrected mental model fits in one sentence: a step count names advances inside one schedule, while runtime depends on dependent updates, work per call, and work around those calls. The boundary matters just as much as the model: the cited sources do not supply a common output, quality target, call mapping, and runtime setup across all three systems. Until they do, the right response to “four steps” is not a ratio but a question: four advances of what?
